# ClawDroid Project Guide

This document is a detailed handoff for future agents and contributors. It was
generated by reading the current repository state, not only the aspirational
docs. Use it as a practical map of what the project is, what technologies it
uses, what advantages the product has, and what gaps need attention.

Last reviewed: 2026-06-14

## One-Sentence Summary

ClawDroid is a native Android AI-agent app that gives the user a transparent,
controllable assistant with a Linux sandbox, command tools, file tools, web
tools, voice, background automation, channels, and Android screen control.

## Product Vision

The product idea is "Codex for your pocket":

- The user chats with an agent on Android.
- The agent can run shell commands inside an embedded Linux runtime.
- The agent can read and write files in a sandbox and shared Documents folder.
- The agent can use connected services and Android-control tools when enabled.
- Every meaningful action is shown inline as collapsible activity steps.
- The user can stop the agent or steer it mid-task.
- The system is local-first: runtime, memory, Room DB, and project sandboxes live
  on the phone.

The source-of-truth product spec is [AGENTS.md](./AGENTS.md). This `project.md`
focuses on the current implementation and how to work on it.

## Core Advantages

### Transparent Agent Work

The main UX advantage is transparency. The agent is not just a chat bubble; it
streams work as inline activity steps. The app stores tool calls and results in
Room and renders them as command/file/web/service steps that can be expanded for
details. This is central to user trust.

### Real Android Native App

The project is Kotlin + Jetpack Compose + Material 3. It is not a web wrapper.
The UI uses native Android services, permissions, notifications, foreground
services, accessibility, speech recognition, MediaProjection, TextToSpeech, and
Room.

### Embedded Linux Runtime

The app bootstraps a Termux-style Linux environment under app-private storage:

```text
/data/data/com.clawdroid.app/files/
  usr/            Linux prefix, bash, apt, libraries
  home/           agent home
  home/projects/ project sandboxes
  home/.memory/  persistent agent memory
  tmp/            temporary runtime files
```

The bootstrap manager downloads a Termux aarch64 ZIP, restores symlinks, patches
shebangs and apt config, applies permissions, and verifies with `bash` plus
`apt update`.

### OpenAI-Compatible Model Interface

The current LLM client targets OpenAI-compatible chat-completions endpoints.
This supports OpenAI, OpenRouter, Groq, Together, SiliconFlow, Ollama-compatible
servers, and other compatible APIs through base URL, API key, and model name.

### Human Control

The app has a stop button while the agent is running and supports steering:
messages typed during a run are queued and injected before the next LLM turn.
External service sending is intended to require user approval.

### Voice and Call Mode

The project includes speech recognition, voice output, an animated voice orb,
live transcript UI, and multiple TTS engines. Android TTS is the fallback.
Piper, OpenAI TTS, ElevenLabs, and Deepgram are implemented. Kokoro and Free
Cloud TTS have engine classes but are not fully routed by `VoiceManager`.

### Android Screen Control

The app can expose Android-control tools to the agent when the accessibility
service is enabled. It can dump the UI tree, tap, swipe, type, navigate back,
open notifications, launch apps, list installed apps, and capture screenshots
when MediaProjection is active.

### Background Work and Channels

Foreground services, WorkManager, notification listener service, channel
interfaces, and heartbeat automation scaffolding exist. The project can draft
WhatsApp replies from notifications and ask for approval before sending.

## App Entry and Navigation

### Current Active Entry Path

- `ClawDroidApplication` initializes `AppConfigManager`.
- `MainActivity` calls `NotificationHelper.ensureChannels()` and
  `AutomationScheduler.schedule()`.
- If `AppConfigManager.ultraAgentEnabled` is true, `ServiceManager.start()` is
  called.
- `MainActivity` uses an enum screen state:
  `Splash`, `Setup`, `Hatching`, `PostSetup`, `Chat`, `Settings`, `Mcp`.
- The active screen switch directly renders `ChatScreen`, `SettingsScreen`, and
  `McpScreen`.

### Alternate Navigation Graph

`ui/navigation/AppNavigation.kt` defines a more complete Navigation Compose
graph with routes for terminal, audio settings, agent settings, automations,
channels, skills, MCP config, and config editor. It is not currently used by
`MainActivity`. This is a major integration point.

### AppContainer Caveat

`core/AppContainer.kt` provides manual dependency access for DB, process
manager, voice manager, and agent engine creation, but `ClawDroidApplication`
does not call `AppContainer.init(context)` right now. The active `ChatScreen`
mostly avoids this by using direct DB and manager construction. The separate
`ChatViewModel` path uses `AppContainer` and would need initialization before
being adopted.

## Data Model

Room database: `data/db/ClawDroidDatabase.kt`

Database name:

```text
clawdroid.db
```

Schema version:

```text
3
```

Entities:

- `ProjectEntity`: project id, name, creation/update times.
- `ConversationEntity`: project id, title, status, cost, token totals, model id,
  optional compaction summary pointer.
- `MessageEntity`: role, content, token count, optional tool call id, optional
  media path and MIME type.
- `ToolCallEntity`: tool name, arguments, result, status, duration.
- `AutomationEntity`: project automation prompt/cron/enabled state.
- `SettingsEntity`: simple key/value table.

DAOs expose Flow-backed project/conversation/message observation, inserts,
updates, deletes, token usage updates, and compaction summary tracking.

Important: the DB uses `fallbackToDestructiveMigration(true)`, so incompatible
schema changes wipe local data unless proper migrations are added.

## Configuration and Secrets

Central config: `core/config/AppConfigManager.kt`

Current storage:

- Main LLM/API/TTS/agent/settings are stored in plain SharedPreferences.
- Channel auth credentials use EncryptedSharedPreferences through
  `core/channels/auth/ChannelAuthManager.kt`.

High-priority security note:

- `AppConfigManager` currently contains hardcoded development credentials.
- Root-level Google OAuth client secret JSON files are present.
- Do not copy those values into docs, logs, screenshots, or prompts.
- Rotate/remove hardcoded keys before any public release.
- Add secret files to `.gitignore` and prefer `local.properties`,
  encrypted preferences, or Android credential flows.

Major settings managed by `AppConfigManager`:

- LLM base URL, API key, model, provider.
- Onboarding and hatching state.
- TTS engine, voice, speed, cloud TTS API keys.
- Agent name, personality, purpose, voice profile, max turns.
- Approval mode, dynamic thinking, emoji tone.
- Active project and conversation ids.
- WhatsApp/SMS/heartbeat toggles.
- MCP toggles and server JSON.
- Owner name and owner info.
- Prompt/config markdown: AGENTS, SOUL, TOOLS, SKILL, CLAUDE.
- Google OAuth client id/secret/refresh token/account email and Gmail/Calendar
  toggles.

## Agent Runtime Flow

Primary classes:

- `core/engine/AgentEngine.kt`
- `core/engine/AgentRunManager.kt`
- `data/api/LlmApiClient.kt`
- `data/api/ContextBuilder.kt`
- `data/api/MessageBuilder.kt`
- `data/api/ToolSchemaRegistry.kt`
- `core/engine/ToolExecutor.kt`

Typical active chat flow:

1. `ChatScreen` inserts a user message into Room.
2. `ChatScreen` starts `AgentRunManager.startRun(...)`.
3. `AgentRunManager` creates an `AgentEngine`.
4. `AgentEngine.run(...)` ensures the Linux bootstrap exists.
5. `AgentEngine` starts configured MCP servers.
6. `ContextBuilder` builds system prompt plus conversation history.
7. Steering messages are drained and inserted before the model call.
8. `LlmApiClient.streamChat(...)` streams text/tool-call deltas.
9. Assistant text and tool calls are saved to Room.
10. Tool calls are executed through `ToolExecutor`.
11. Tool result messages are saved with matching tool call ids.
12. Loop repeats until no tool calls remain, max turns are reached, or stop is
    requested.
13. A session summary is appended to `home/.memory/memory.md`.

System prompt assembly:

- Uses `AppConfigManager.agentName`, personality, purpose, owner info.
- Reads project-specific `agent.md` or `AGENT.md` from
  `files/home/projects/<projectId>/`.
- Injects persistent memory from `MemoryManager`.
- Adds Android screen-control instructions if accessibility service is active.
- Instructs the agent not to auto-send messages to external services without
  user-specified content.

## LLM Client

Class: `data/api/LlmApiClient.kt`

Behavior:

- Calls `<baseUrl>/chat/completions`.
- Uses streaming SSE.
- Sends OpenAI-compatible `messages`, `tools`, `tool_choice`.
- Accumulates fragmented tool call JSON by tool-call index.
- Emits text deltas, tool-call streaming updates, completed tool calls, usage,
  errors, and done.
- Supports image attachment input by embedding files as base64 `image_url`
  data URLs when a message has `mediaPath` and `mediaMimeType`.

Headers:

- `Authorization: Bearer <apiKey>`
- `Content-Type: application/json`
- `Accept: text/event-stream`
- OpenRouter-style `HTTP-Referer` and `X-Title`.

Current limitations:

- `RetryClient` exists but is not wired into `LlmApiClient`.
- Retry-after parsing is not implemented in the active streaming path.
- Network reconnect/queue behavior from `AGENTS.md` is aspirational.
- Usage is recorded when the provider returns it; otherwise token estimates are
  used.

## Terminal and Process Runtime

Key files:

- `core/bootstrap/BootstrapManager.kt`
- `core/bootstrap/EnvironmentSetup.kt`
- `core/terminal/ProcessManager.kt`
- `core/terminal/ManagedProcess.kt`
- `core/terminal/OutputBuffer.kt`
- `core/terminal/InputTranslator.kt`
- `core/terminal/AnsiStripper.kt`

Current behavior:

- Commands run through `ProcessBuilder`:
  `files/usr/bin/bash --noprofile --norc -c <command>`.
- Environment variables are cleared and replaced with `EnvironmentSetup.values`.
- `executeCommand` waits for completion up to timeout and errors on timeout.
- `startProcess` returns a process id after a 2.5 second initial wait.
- Output is merged stdout/stderr, ANSI-stripped, and kept in a 500-line ring
  buffer for tracked processes.
- Prompt detection looks for Y/n prompts, password prompts, Python prompt, and
  inquirer-style prompts.
- `InputTranslator` supports `[ENTER]`, `[TAB]`, arrows, `[CTRL+C]`,
  `[CTRL+D]`, `[CTRL+Z]`, `[HOME]`, `[END]`, `[BACKSPACE]`.

Important mismatch with product spec:

- Current code does not use a PTY or `script -q -c`.
- `execute_command` does not auto-promote timed-out commands to background.
- `killProcess` calls `destroyForcibly()` on the parent process, not a process
  group kill.
- `AgentEngine.stop()` sets a flag but does not directly call
  `ProcessManager.killAll()`.
- Tool calls are executed serially in `AgentEngine`, not parallelized.

## Filesystem Boundaries

`FileTools.resolveAgentPath(...)` restricts file access to:

- Agent home: `files/home`
- Linux prefix: `files/usr`
- Temp: `files/tmp`
- Shared folder: `/storage/emulated/0/Documents/ClawDroid/`

Shared folder manager creates:

```text
/storage/emulated/0/Documents/ClawDroid/
  Inbox/
  Output/
  Projects/
  Exports/
```

For external storage paths, file tools try to launch Android settings for all
files access and poll briefly for permission.

## Memory and Context Compaction

Persistent memory:

- `core/memory/MemoryManager.kt`
- File path: `files/home/.memory/memory.md`
- Created on first use with known facts.
- Appends session summaries after agent runs.
- Injected into the system prompt through `MessageBuilder`.

Compaction:

- `core/engine/CompactionManager.kt`
- Default token limit is 128,000.
- Compaction threshold is 80 percent of limit.
- Uses the active `LlmApiClient` to summarize the full transcript.
- Saves summary as an assistant message beginning with `[Compacted Summary]`.
- Conversation stores `summaryMessageId`.

Current limitations:

- Old messages are not archived/deleted; summary pointer controls what is sent.
- UI hides compacted summary messages but does not show a dedicated
  "context compacted" divider.
- Compaction uses the active client/model, not a cheaper configured model.

## Known Current Gaps and Risks

### Security

- Hardcoded dev LLM key and Google client secret exist in source/config.
- Root client secret JSON files exist.
- Main `AppConfigManager` uses plain SharedPreferences for API keys.
- `.gitignore` does not appear to cover the root client secret JSON files.

### Navigation

- `MainActivity` does not use `AppNavHost`, so many route-specific screens are
  present but not reachable through the active screen system.
- Sidebar callbacks for Audio, Automations, Channels, Skills, Agent Config, and
  Terminal mostly route to general Settings or show a placeholder toast in the
  active `ChatScreen`.

### Terminal and Processes

- No real PTY yet.
- No process group tracking/kill.
- No timeout-to-background promotion for `execute_command`.
- Stop does not guarantee all running shell processes are killed.
- Command execution has two paths: `CommandTool` and `ProcessManager`; behavior
  is not perfectly unified.

### Agent Loop

- Tool calls are serial, not parallel.
- Approval modes are stored but not consistently enforced in `ToolExecutor`.
- Cost limits are not enforced.
- `RetryClient` is not wired into active API calls.
- Some steering persistence can be duplicated because `ChatScreen` saves the
  steering message and `AgentEngine` also drains/saves steering.

### UI/Data Duplication

- Active `ChatScreen`, `ChatViewModel`, `AgentRunManager`, and `AgentEngine` all
  contain overlapping chat/run/persistence concepts.
- Future refactors should choose one active architecture and remove or adapt the
  unused path carefully.

### Web

- `browse_web` uses raw HTTP fetch and regex HTML stripping, not a WebView with
  JavaScript extraction as described in the product spec.
- `web_search` scrapes DuckDuckGo HTML with regex and returns up to five
  results.

### Testing

- No unit tests or instrumented tests currently exist.
- High-risk classes needing tests first: `InputTranslator`, `OutputBuffer`,
  `DefensiveJsonParser`, `MessageBuilder`/`ContextBuilder`, `LoopDetector`,
  `ToolExecutor`, `LlmApiClient` SSE parsing, `FileTools.resolveAgentPath`.

### Store Compatibility

- `targetSdk = 28` is intentional for executable writable storage compatibility,
  but it blocks normal Google Play distribution and may trigger modern Android
  policy friction.

## Important Product Rules for Future Agents

Follow these rules when modifying the project:

- Preserve transparency: every agent action should be visible as an activity
  step or notification.
- Preserve user control: stop and steering are not optional.
- Do not auto-send external messages unless the user has supplied the exact
  content or the configured trust mode explicitly allows it and code enforces it.
- Prefer non-interactive shell flags (`-y`, `--yes`, `--quiet`) over interactive
  prompts.
- Do not use terminal editors for agent file writes; use file tools or direct
  file APIs.
- Keep sandbox and shared-folder boundaries explicit.
- Do not copy or expose API keys from source or local files.

## Glossary

- Agent: the LLM-driven assistant loop.
- Activity step: an inline, collapsible record of a tool/action.
- Sandbox: app-private Linux environment under `files/usr` and `files/home`.
- Shared folder: user-visible Documents/ClawDroid tree.
- Steering: user message entered while the agent is running.
- MCP: Model Context Protocol subprocess connector exposing extra tools.
- Heartbeat: recurring background task based on project heartbeat files.
- Ultra Agent Mode: foreground/background mode with broader device automation.

## Final Context for Agents

This project is already much more than a demo chat app: it has a real Android
UI, a Linux bootstrap, an LLM tool loop, Room persistence, voice, services,
notifications, channels, MCP scaffolding, and Android-control tools. The main
work now is consolidation and hardening. Be careful not to build parallel
systems on top of parallel systems. First determine which implementation path is
active, then tighten it.

When in doubt, preserve the core product promise:

```text
The user delegates work, sees what happened, can stop it immediately, and stays
in control.
```
